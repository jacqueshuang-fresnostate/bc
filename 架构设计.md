后端使用rust去进行开发，管理员的前端使用react + vite +tailwindcss + semi-ui
现在需要设计一个彩票的网站，他的具体功能有 管理后台，手机端（手机端后续再进行）
### 管理后台的功能
#### 公共功能
1. 添加用户，用户信息维护，用户资金查看
2. 订单管理
3. 财务管理
4. 在线客服
5. 管理员管理
6. 角色权限管理
7. 系统设置
#### 主要功能
需要依据 3个号码玩法规则说明.md 和 5个玩法规则说明.md，开设彩种
##### 彩种需求说明
1. 彩种有几种开奖的模式
   > 平台开奖(就是可以配置开奖号码由平台去随机号码生成)
   > API 接口 （就是通过API接口去请求开奖结果）
   > 控制指定号码（就是可以配置某个彩种某个期号，开奖是 特定的号码）
2. 彩种的开奖时间
   > 周期开奖 有比如60秒一个周期 5分钟一个周期的 周期开奖
   > 每日固定时间开奖 有比如 每天的21:00:15 开奖
   > 周开奖 比如周二 周四 晚上 9点开奖
3. 开奖的模式的多彩种使用
   > 比如福彩3d的API开奖请求结果，可以给排列3 作为开奖结果
   > 平台自动生成的某个生成器，也可以给多个彩种作为开奖结果
 ##### 合买需求说明
 每个彩种都可以开启关闭合买功能
 1. 每个彩种可以配置彩种合买的份数的最低金额 （比如发起人发起1000块的合买计划，那么设置最低合买金额是1块钱，那么这个合买计划最多就只能分成1000份）
 2. 最低起购配置 合买计划可以配置发起人的最低起购比例 （比如配置了10%的最低起购，那么发起人在发起1000块的合买计划的时候他需要自己，最低购买100块钱）
 3. 可以配置参与人最低的购买金额 (比如发起人发起1000块的合买计划，配置了10块钱的最低起购，用户看到这个计划的时候购买最低得购买10块钱，占1% 的股份 ，后续中奖按照比例去进行分配彩金)
#### 机器人
1. 合买机器人
   > 在某个彩种开盘期间可以不断的发起合买计划
   > 在非机器人发起合买计划的以后可以让这个合买计划满单（就是到达合买的金额）
2. 购彩机器人
   > 可以在某个彩种开盘期间去购买彩种，模拟用户购买的操作
#### 用户注册
用户注册可以使用用户名注册，也可以使用邮箱注册（需要后台开启邮箱注册功能）
#### 邀请
用户分为普通用户，和代理，只有代理可以邀请，并且邀请可以进行返利
#### 返利
1. 代理邀请下级，下级有充值返利给上级，管理员可以配置返利的比例
   > 返利有立即返利的，也有充值阶梯返利的 ，可以后台切换

## 2026-06-02 彩种数据库持久化范围

本阶段为彩种管理增加 PostgreSQL 持久化能力，但保留无数据库环境下的内存模式，避免本地开发、测试和前端预览被外部数据库阻塞。

### 后端新增范围

1. 新增 `DATABASE_URL` 运行模式：未配置时使用内存彩种仓储；配置后连接 PostgreSQL、运行迁移并使用数据库仓储。
2. 新增 `backend/migrations/20260602150315_create_lotteries.sql`，创建 `lotteries` 表。
3. 新增统一彩种仓储入口，路由层不区分内存仓储和数据库仓储。
4. 数据库为空时自动写入种子彩种；数据库已有彩种时不覆盖已有数据。
5. 当前 Rust 工具链为 `1.92.0`，SQLx `0.9.0` 要求 Rust `1.94.0`，因此本阶段使用兼容当前工具链的 SQLx `0.8.6`。

### 数据表范围

`lotteries` 表保存彩种基础配置：

1. `id`、`name`、`number_type`、`draw_mode`、`sale_enabled` 使用明确列。
2. `schedule`、`group_buy`、`play_categories` 暂用 `jsonb` 保存，保持现有前后端契约。
3. `created_at`、`updated_at` 用于后续审计和排序。

### 后续仍需实现

1. 增加数据库容器或环境配置脚本，方便本地完整数据库联调。
2. 根据开奖时间、玩法和合买配置稳定程度，评估是否从 `jsonb` 拆分为独立关系表。
3. 彩种删除后需要增加订单、开奖历史和机器人任务的删除保护。

## 2026-06-02 玩法规则引擎范围

本阶段把“玩法规则”从后台占位入口升级为可验证的真实能力。玩法规则引擎是后续投注订单、开奖计奖、派奖、合买分成、机器人购彩和风控校验的基础，因此规则计算统一放在后端服务层，管理后台只负责录入参数和展示结果。

### 后端新增范围

1. 新增玩法规则领域模型，覆盖玩法代码、选号结构、大小单双属性、规则目录和评估结果。
2. 新增玩法规则服务，支持注数计算、投注展开和中奖判断。
3. 支持 3 位玩法：直选、组三复式、组三胆拖、组六复式、组六胆拖。
4. 支持 5 位玩法：前 3 / 中 3 / 后 3 直选、直选组合、组三复式、组三胆拖、组六复式、组六胆拖、大小单双。
5. 5 位大小单双当前默认按后两位判断，十位和个位分别支持“大、小、单、双”属性。
6. 新增接口：
   - `GET /api/admin/play-rules`
   - `POST /api/admin/play-rules/evaluate`
7. 服务层校验开奖号码长度、数字范围、玩法与号码类型匹配、直选位数、胆码/拖码重复和必要选号参数。

### 管理后台新增范围

1. 新增“玩法规则”真实页面，替换原有占位页。
2. 页面支持选择 3 位和 5 位玩法、填写开奖号码、填写按位选号、复式选号、胆拖选号和大小单双属性。
3. 页面调用后端评估接口展示注数、展开投注、命中状态和命中投注。
4. 侧边栏“玩法规则”入口加入独立图标，并由 dashboard 模块目录直接进入真实页面。

### 后续仍需实现

1. 订单创建需要接入玩法规则引擎，保存标准化投注明细、注数和投注金额。
2. 开奖源与期号管理需要接入平台开奖、API 开奖和指定号码，并把开奖结果传给计奖流程。
3. 计奖与派奖需要基于规则引擎判断命中，再按赔率、奖金表、合买份额和资金流水结算。
4. 大小单双的开奖位置后续需要做成可配置项，适配不同彩种或平台规则差异。
5. 规则引擎后续需要增加赔率、奖金上限、封盘校验、异常订单保护和审计日志。
6. 手机端购彩页面需要复用同一套后端规则，不允许只在前端自行计算注数和中奖。

## 2026-06-02 订单与投注基础范围

本阶段把“订单管理”从后台占位入口升级为可创建、可查询、可取消的基础投注订单能力。订单创建会复用后端玩法规则引擎计算注数、展开投注和订单金额，为后续开奖、计奖、派奖、财务流水和机器人购彩提供真实订单数据入口。

### 后端新增范围

1. 扩展订单领域模型，新增创建订单请求、订单详情、订单摘要和订单状态。
2. 新增内存订单仓储，支持创建订单、订单列表、订单详情、取消待开奖订单和最近订单摘要。
3. 新增订单接口：
   - `GET /api/admin/orders`
   - `GET /api/admin/orders/{id}`
   - `POST /api/admin/orders`
   - `PATCH /api/admin/orders/{id}/cancel`
4. 创建订单时从彩种仓储读取彩种配置，并校验彩种存在、销售状态开启、玩法号码类型匹配、彩种已启用对应玩法分类。
5. 创建订单时调用玩法规则引擎展开投注，后端计算 `stakeCount`、`unitAmountMinor` 和 `amountMinor`，不接受前端传入最终金额。
6. dashboard 的今日订单数量和最近订单改为读取订单仓储，不再使用静态订单演示函数。

### 管理后台新增范围

1. 新增“订单管理”真实页面，支持查看订单列表、订单状态、彩种、期号、玩法、注数和金额。
2. 页面支持创建测试投注单，填写用户 ID、彩种、期号、玩法、选号参数和单注金额。
3. 页面支持取消待开奖订单。
4. 工作台新增“最近订单”区块，订单创建后可以看到最近订单回流。

### 后续仍需实现

1. 订单仓储需要增加 PostgreSQL 持久化、迁移和查询筛选。
2. 订单创建暂不扣减用户余额，后续必须接入用户资金、冻结金额和资金流水。
3. 后续需要接入开奖期号、封盘时间、开奖结算、计奖、派奖和订单状态流转。
4. 机器人购彩需要复用同一套订单创建接口，不能跳过订单校验。
5. 合买订单、份额拆分和中奖分配仍需单独实现。
