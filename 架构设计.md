现在需要设计一个彩票的网站，他的具体功能有 管理后台，手机端（手机端后续再进行）
### 管理后台的功能
#### 公共功能
1. 添加用户，用户信息维护，用户资金查看
2. 订单管理
3. 财务管理
4. 在线客服
5. 管理员管理
6. 角色权限管理
7. 系统设置
#### 主要功能
需要依据 3个号码玩法规则说明.md 和 5个玩法规则说明.md，开设彩种
##### 彩种需求说明
1. 彩种有几种开奖的模式
   > 平台开奖(就是可以配置开奖号码由平台去随机号码生成)
   > API 接口 （就是通过API接口去请求开奖结果）
   > 控制指定号码（就是可以配置某个彩种某个期号，开奖是 特定的号码）
2. 彩种的开奖时间
   > 周期开奖 有比如60秒一个周期 5分钟一个周期的 周期开奖
   > 每日固定时间开奖 有比如 每天的21:00:15 开奖
   > 周开奖 比如周二 周四 晚上 9点开奖
3. 开奖的模式的多彩种使用
   > 比如福彩3d的API开奖请求结果，可以给排列3 作为开奖结果
   > 平台自动生成的某个生成器，也可以给多个彩种作为开奖结果
 ##### 合买需求说明
 每个彩种都可以开启关闭合买功能
 1. 每个彩种可以配置彩种合买的份数的最低金额 （比如发起人发起1000块的合买计划，那么设置最低合买金额是1块钱，那么这个合买计划最多就只能分成1000份）
 2. 最低起购配置 合买计划可以配置发起人的最低起购比例 （比如配置了10%的最低起购，那么发起人在发起1000块的合买计划的时候他需要自己，最低购买100块钱）
 3. 可以配置参与人最低的购买金额 (比如发起人发起1000块的合买计划，配置了10块钱的最低起购，用户看到这个计划的时候购买最低得购买10块钱，占1% 的股份 ，后续中奖按照比例去进行分配彩金)
#### 机器人
1. 合买机器人
   > 在某个彩种开盘期间可以不断的发起合买计划
   > 在非机器人发起合买计划的以后可以让这个合买计划满单（就是到达合买的金额）
2. 购彩机器人
   > 可以在某个彩种开盘期间去购买彩种，模拟用户购买的操作
#### 用户注册
用户注册可以使用用户名注册，也可以使用邮箱注册（需要后台开启邮箱注册功能）
#### 邀请
用户分为普通用户，和代理，只有代理可以邀请，并且邀请可以进行返利
#### 返利
1. 代理邀请下级，下级有充值返利给上级，管理员可以配置返利的比例
   > 返利有立即返利的，也有充值阶梯返利的 ，可以后台切换

1. **目标**：手机端需要和后端建立当前系统自己的实时事件链路，不能继续使用旧系统残留的 `/ws/lottery`。

## 2026-06-04 Docker 镜像后端 502 修复

1. **目标**：单镜像部署后，请求后端接口不能因为后端启动失败而只返回 Nginx 502；容器需要能清楚暴露后端启动失败原因。
2. **实现方式**：
   - Docker 入口脚本启动后端后，先等待 `http://127.0.0.1:${BACKEND_PORT}/api/health` 健康检查通过，再启动 Nginx。
   - 新增 `BACKEND_STARTUP_TIMEOUT_SECONDS` 配置，默认等待 60 秒，避免数据库迁移或初始化稍慢时误判。
   - 后端启动失败或运行中退出时，入口脚本会关闭 Nginx 并让容器退出，避免页面正常但接口长期 502。
   - Docker healthcheck 的 `start-period` 调整为 60 秒，和入口脚本等待窗口一致。
   - 后端启动前会校验 `DATABASE_URL`，非空时必须以 `postgres://` 或 `postgresql://` 开头；格式错误时输出中文配置错误，不再出现 `RelativeUrlWithoutBase`。
   - 后端会先完成数据库和路由初始化，再输出“后台接口服务已开始监听”，避免启动失败时误判为已监听。
3. **验收标准**：
   - 镜像启动后 `/api/health` 能返回 200。
   - 后端连接数据库失败时，容器应失败退出并输出中文日志，而不是继续返回 502。
   - `DATABASE_URL` 缺少 PostgreSQL 协议前缀时，应输出明确中文错误。

## 2026-06-03 Docker 单镜像部署范围

本阶段新增前后端同项目 Docker 单镜像部署方案。镜像构建时分别编译管理后台静态资源和 Rust 后端二进制，运行时由 Nginx 服务前端页面，并把 `/api/` 反向代理到同容器内的后端服务。

## 2026-06-03 API68 周期彩种期号时间对齐修正

## 2026-06-05 容器 WebSocket 开奖推送修复

3. 手机端 `useWebSocket` 已经连接 `/api/user/realtime`，`LayoutView` 会把 `lastMessage` 通过 `ws-message` 传给当前页面。
4. 手机端首页和下注页已经监听开奖事件，命中当前彩种后会刷新最近开奖或下注配置。

### 后续仍需实现

1. 发布新镜像后需要做一次容器级 WebSocket 验证：连接 `/api/user/realtime`，执行手动开奖或等待调度开奖，确认收到 `lottery.draw_result`。
2. 后续可在系统设置或调度页增加实时连接诊断，显示最近一次公开事件时间、连接数和代理升级状态。
3. 后续可增加自动化端到端测试，覆盖“容器 Nginx 代理 -> WebSocket 连接 -> 后台手动开奖 -> 手机端收到开奖事件”的完整链路。

## 2026-06-05 合买机器人按玩法随机选号

本阶段修正合买机器人自动发起计划时投注内容固定的问题。机器人不再长期使用 `1|2|3` 这类样例投注，而是按机器人、彩种、期号、玩法和玩法顺序派生随机选号内容。

### 后端调整范围

1. 合买机器人自动发起计划时，根据玩法生成合法投注文本：
   - 直选玩法生成 `百位|十位|个位` 三段数字，例如 `4|0|8`。
   - 直选组合和组选复式生成不重复数字列表，例如 `2,7,9`。
   - 胆拖玩法生成 `胆码|拖码`，并保证胆码与拖码不重叠。
   - 大小单双生成 `tens:属性|ones:属性`，例如 `tens:big|ones:odd`。
2. 随机选号使用业务上下文派生确定性随机种子，同一期机器人计划可复现，不同期号和玩法会变化，便于测试和排障。
3. 生成后的投注内容仍必须通过 `parse_group_buy_selection` 解析，并继续调用订单报价校验，确保不会绕过玩法规则。

### 验证要求

1. 后端需要覆盖所有当前支持玩法生成的机器人投注内容均可解析、可展开。
2. 后端需要覆盖直选投注内容会随期号变化，不能一直固定为 `1|2|3`。
3. 后端需要通过 `cargo fmt`、`cargo check` 和 `cargo test`。
