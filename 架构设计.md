现在需要设计一个彩票的网站，他的具体功能有 管理后台，手机端（手机端后续再进行）
### 管理后台的功能
#### 公共功能
1. 添加用户，用户信息维护，用户资金查看
2. 订单管理
3. 财务管理
4. 在线客服
5. 管理员管理
6. 角色权限管理
7. 系统设置
#### 主要功能
需要依据 3个号码玩法规则说明.md 和 5个玩法规则说明.md，开设彩种
##### 彩种需求说明
1. 彩种有几种开奖的模式
   > 平台开奖(就是可以配置开奖号码由平台去随机号码生成)
   > API 接口 （就是通过API接口去请求开奖结果）
   > 控制指定号码（就是可以配置某个彩种某个期号，开奖是 特定的号码）
2. 彩种的开奖时间
   > 周期开奖 有比如60秒一个周期 5分钟一个周期的 周期开奖
   > 每日固定时间开奖 有比如 每天的21:00:15 开奖
   > 周开奖 比如周二 周四 晚上 9点开奖
3. 开奖的模式的多彩种使用
   > 比如福彩3d的API开奖请求结果，可以给排列3 作为开奖结果
   > 平台自动生成的某个生成器，也可以给多个彩种作为开奖结果
 ##### 合买需求说明
 每个彩种都可以开启关闭合买功能
 1. 每个彩种可以配置彩种合买的份数的最低金额 （比如发起人发起1000块的合买计划，那么设置最低合买金额是1块钱，那么这个合买计划最多就只能分成1000份）
 2. 最低起购配置 合买计划可以配置发起人的最低起购比例 （比如配置了10%的最低起购，那么发起人在发起1000块的合买计划的时候他需要自己，最低购买100块钱）
 3. 可以配置参与人最低的购买金额 (比如发起人发起1000块的合买计划，配置了10块钱的最低起购，用户看到这个计划的时候购买最低得购买10块钱，占1% 的股份 ，后续中奖按照比例去进行分配彩金)
#### 机器人
1. 合买机器人
   > 在某个彩种开盘期间可以不断的发起合买计划
   > 在非机器人发起合买计划的以后可以让这个合买计划满单（就是到达合买的金额）
2. 购彩机器人
   > 可以在某个彩种开盘期间去购买彩种，模拟用户购买的操作
#### 用户注册
用户注册可以使用用户名注册，也可以使用邮箱注册（需要后台开启邮箱注册功能）
#### 邀请
用户分为普通用户，和代理，只有代理可以邀请，并且邀请可以进行返利
#### 返利
1. 代理邀请下级，下级有充值返利给上级，管理员可以配置返利的比例
   > 返利有立即返利的，也有充值阶梯返利的 ，可以后台切换

## 2026-06-03 Docker 单镜像部署范围

本阶段新增前后端同项目 Docker 单镜像部署方案。镜像构建时分别编译管理后台静态资源和 Rust 后端二进制，运行时由 Nginx 服务前端页面，并把 `/api/` 反向代理到同容器内的后端服务。

### 镜像结构

1. `admin-build` 阶段使用 Node 执行 `npm ci` 和 `npm run build`，产物为 `admin/dist`。
2. `backend-build` 阶段使用 Rust 执行 `cargo build --release`，产物为 `bc-backend`。
3. 运行阶段基于 Nginx，复制前端静态资源、后端二进制、Nginx 配置和启动脚本。
4. 容器内后端通过 `BACKEND_PORT` 监听，默认 `8080`；Nginx 对外监听 `80`。
5. `/api/` 请求由 Nginx 按 `BACKEND_PORT` 动态渲染后的端口反向代理到容器内后端，其它请求走前端 SPA fallback。
6. Docker healthcheck 请求 `http://127.0.0.1/api/health`。

### 使用范围

1. 根目录 `Dockerfile` 负责构建单镜像。
2. `docker/nginx.conf` 负责前端静态文件和 API 反向代理。
3. `docker/entrypoint.sh` 同时启动后端和 Nginx。
4. `docker-compose.yml` 提供本地一键构建运行入口，并同时启动 PostgreSQL。
5. `部署说明.md` 记录构建、运行和环境变量。

### 数据库接入范围

1. 应用镜像本身不内置 PostgreSQL；单独 `docker run` 时如果不传 `DATABASE_URL`，后端仍使用内存演示数据。
2. `docker compose up --build` 会启动 `postgres:16-alpine`，创建 `bc_postgres_data` volume，并把应用 `DATABASE_URL` 指向 Compose 内的 PostgreSQL。
3. Compose 默认对外暴露 `8080`，可通过 `APP_PORT=18080 docker compose up --build` 覆盖宿主机端口。
4. 后端检测到 `DATABASE_URL` 后会运行 SQLx migrations，并让彩种管理、玩法赔率配置使用 PostgreSQL `lotteries` 表。
5. 用户、管理员、角色权限、系统设置、注册配置、管理员会话、订单、结算批次、开奖期号、开奖源、彩种控制台控制号码、资金账户、资金流水、合买计划、邀请关系、返利策略、机器人配置、客服会话、调度配置和调度运行历史使用独立业务表持久化。
6. 数据库为空时各业务表会写入原有种子数据；数据库已有业务数据时优先恢复数据库状态，不会被种子数据覆盖。

1. `ApiDrawSourceLatestIssue` 新增 `drawTime` 语义数据，API68 最新期号解析时同时读取 `result.data[0].preDrawTime`。
2. API68 周期彩种生成计划时，使用 `preDrawTime + 周期间隔 * 期号偏移` 计算下一期 `scheduledAt`，而不是直接使用服务器当前秒数。
3. 如果本地已经存在同彩种更大的 API 期号，后端按本地最大期号与外部最新期号的差值计算时间偏移，避免已有错位期号继续把后续期号带偏。
4. 生成计划会跳过 `saleClosedAt <= now` 的候选期号，并同步递增期号，避免创建已经封盘却显示为 `open` 的期号。
5. 福彩 3D、排列 3 这类每日开奖彩种继续按后台彩种配置的每日开奖时间生成，但期号仍使用 API68 最新 `preDrawIssue` 递增。

### 本地验证结果

1. 使用本地 `18120` 后端确认调度启用前 `draw-scheduler/status` 为 `enabled=false`，期号列表为空；通过后台配置接口启用后，调度生成了福彩 3D、排列 3、澳洲 5 分彩和 60 秒时时彩的 open 期号。
2. 60 秒时时彩完成一轮封盘、开奖，并自动补出下一期 open 期号，证明常驻调度的“开奖后继续开盘下一期”链路可运行。
3. `cargo test` 覆盖澳洲 5 分彩以 `preDrawTime=2026-06-03 11:18:40` 生成 `51320850 / 2026-06-03 11:23:40`，并覆盖封盘后跳到 `51320851 / 2026-06-03 11:28:40`。

### 后续仍需实现

1. 开奖源配置页面后续可显示 API68 最新 `preDrawIssue`、`preDrawTime` 和本地下一期开奖时间，方便运营直接判断来源是否延迟。
2. 调度运行历史后续需要展示跳过彩种和跳过期号的明细原因，而不仅是数量。
3. API 开奖源后续可增加供应商时间延迟阈值、重试间隔和异常告警。

## 2026-06-03 腾讯分分彩 KJAPI 彩种接入

本阶段按新开奖接口 `https://kjapi.net/hall/hallajax/getLotteryInfo?lotKey=txffc` 接入腾讯分分彩。该接口不是 API68 格式，响应数据位于 `result.data` 对象中，已开奖期号和开奖号码为 `preDrawIssue`、`preDrawCode`、`preDrawTime`，下一期为 `drawIssue`、`drawTime`。

### 后端调整范围

1. 新增彩种 `txffc`，名称为“腾讯分分彩”，号码类型为 5 位，开奖模式为 API，开奖周期为 60 秒，默认销售开启。
2. 新增开奖源供应商 `kjApi`，默认来源 `kj-txffc` 绑定 `txffc`，endpoint 为 `https://kjapi.net/hall/hallajax/getLotteryInfo`，`lotCode` 字段复用保存 `lotKey=txffc`。
3. KJAPI 开奖解析按 `result.data.preDrawIssue` 匹配后台期号，并使用 `preDrawCode` 保存开奖号码；开奖号码仍统一使用英文逗号分隔格式。
4. KJAPI 最新期号解析同时读取 `drawIssue` 和 `drawTime`，生成下一期时优先使用供应商返回的下一期开奖时间；如果该候选期已经过封盘时间，会递增到下一期。
5. API 期号序列从 32 位整数升级为 64 位整数，支持 `202606031179` 这类 12 位期号。
6. PostgreSQL 模式下，彩种和开奖源初始化改为补齐缺失默认项：已有同 ID 配置不会被覆盖，缺失的 `txffc` 和 `kj-txffc` 会在服务重启后写入数据库。

### 管理后台调整范围

1. 开奖源供应商类型新增 `kjApi`，开奖源配置表单新增 `KJAPI` 选项。
2. 开奖源配置表单的 `lotCode` 标签调整为 `lotCode / lotKey`，支持 API68 数字 `lotCode` 和 KJAPI 字符串 `lotKey`。
3. 开奖源预设新增“腾讯分分彩采集”，默认填入 `kj-txffc`、`KJAPI-腾讯分分彩`、`txffc` 和 `https://kjapi.net/hall/hallajax/getLotteryInfo`。

### 后续仍需实现

1. 后台开奖源页面后续可增加“测试连接”按钮，直接展示 KJAPI 当前 `preDrawIssue`、`drawIssue`、`serverTime` 和解析后的本地下一期开奖时间。
2. 调度历史后续需要展示 KJAPI 请求失败、期号不匹配和供应商延迟的明细原因。
3. 如果后续继续接入非 API68 来源，应抽象供应商解析器注册表，减少 `draw_api.rs` 中的分支增长。

## 2026-06-03 开奖等待原因可视化与控制台当前期修正

本阶段排查“彩种控制台到达开奖时间后一直显示等待开奖”的问题。实际复现中，`txffc` 存在旧期 `202606031202` 已封盘且到期开奖，但 KJAPI 当前返回期号已经跳到后续期号，因此自动开奖无法从最新接口拿到旧期开奖号码，只能跳过并保留待补开奖状态。此前调度历史只保存跳过数量，控制台又优先展示最早的 `open/closed` 期号，导致旧的待补期号一直压住新开盘期号。

### 后端调整范围

1. `draw_scheduler_runs` 新增 `skipped_issues` 和 `skipped_lotteries` 两个 `jsonb` 字段，保存每轮调度跳过的期号和彩种明细。
2. `DrawSchedulerRunRecord` 返回跳过明细，后台可以直接看到跳过期号、彩种和原因。
3. 自动开奖保存跳过原因时使用中文业务前缀，不再把 `not found:` 这类英文错误前缀暴露给后台页面。
4. API 开奖源未找到期号时，错误原因会带上“当前返回期号”，例如 `API 开奖源未找到期号 202606031202 的开奖号码，当前返回期号 202606031211`。
5. 调度跳过停售彩种时使用中文原因“彩种销售已关闭”。

### 管理后台调整范围

1. “开奖期号与开奖源”的常驻调度卡片展示最近一轮跳过期号和跳过彩种的明细原因。
2. 彩种控制台顶部展示调度启停状态和执行周期。
3. 彩种控制台的“开奖倒计时”到点后按状态显示“等待开奖源”“等待调度”或“调度已关闭”，避免只有“等待开奖”一个模糊状态。
4. 彩种控制台选择当前期时优先展示 `open` 期号；如果存在旧的 `closed` 待补开奖期号，则通过“待补开奖 N”标签提示，不再让旧漏开奖期号长期压住新的开盘期。

### 本地验证结果

1. 使用本地 `18121` 后端连接外部 PostgreSQL，确认调度状态接口返回 `skippedIssues/skippedLotteries` 明细。
2. 复现 `txffc` 旧期 `202606031202` 被 KJAPI 越过的情况，接口返回原因包含当前返回期号 `202606031211`。
3. 当前 `txffc` 同时存在旧的 `closed` 待补期开奖和新的 `open` 期号，控制台会优先展示新 open 期，并用待补开奖标签提示旧期。

### 后续仍需实现

1. 开奖源配置页后续需要增加“测试连接”和“查看最新外部期号”，运营可以提前发现外部源是否跳期或延迟。
2. 对外部 API 已经越过且无法补取号码的旧期，后续需要增加管理员复核入口，可选择手动录入开奖号码、取消期号或标记异常。
3. 调度历史后续可增加原始响应留痕、失败重试次数和告警阈值。
