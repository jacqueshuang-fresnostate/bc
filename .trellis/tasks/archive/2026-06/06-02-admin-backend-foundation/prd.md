# 基础工程与管理后台 MVP 骨架

## 目标

根据 `架构设计.md` 中的彩票系统方向，先搭建可运行的 Rust 后端与 React 管理后台基础工程，为后续用户、订单、财务、客服、彩种、开奖、合买、机器人、邀请返利等功能提供稳定的模块边界和开发入口。

## 已知信息

- 后端使用 Rust 开发。
- 管理员前端使用 React + Vite + Tailwind CSS + Semi UI。
- 当前优先建设管理后台，手机端后续再做。
- 彩种和玩法需要依据 `3个号码玩法规则说明.md`、`5个玩法规则说明.md` 建模。
- `架构设计.md` 的 `# 后续` 当前为空，没有新增功能条目。
- 当前项目尚无真实代码，只有需求文档、Trellis 配置和规则说明。
- `TODO.md` 不存在，需要本轮创建并持续记录开发进展。
- 项目文档需要使用中文输出，代码标识、命令、路径、第三方库名和协议字段名可以保留英文。

## 临时假设

- 本轮采用 Axum + Tokio 搭建 Rust API 服务。
- 本轮不接入真实数据库，先提供领域模型、API 路由、静态示例数据与统一响应结构；数据库迁移和持久化放到下一阶段。
- 管理后台本轮先实现导航、页面骨架、统计概览和模块入口，不实现完整业务提交流。
- 彩票规则本轮进入类型与配置骨架，不实现完整计奖、派奖和风控流程。

## 需求

- 创建后端工程目录，提供可运行的 Rust API 服务。
- 后端需要包含健康检查接口、统一响应结构、错误结构和基础路由模块。
- 后端需要建立核心领域模型：用户、管理员、角色权限、彩种、开奖模式、开奖时间、合买配置、订单、财务、机器人、代理返利。
- 后端需要提供管理后台首批只读 API，支撑前端展示模块入口和基础示例数据。
- 创建管理后台工程目录，使用 React + Vite + Tailwind CSS + Semi UI。
- 管理后台需要具备应用布局、侧边导航、概览页和各核心模块页面入口。
- 管理后台需要调用后端 API 获取概览数据，并在接口不可用时保持清晰的错误状态。
- 更新 `架构设计.md`，记录首期工程约定和本轮实现范围。
- 创建并维护 `TODO.md`，记录每次完成的任务、解决的问题和时间。

## 验收标准

- [x] `backend` 可以通过 `cargo check`。
- [x] `backend` 暴露 `/api/health` 与管理后台概览相关接口。
- [x] `admin` 可以通过 `npm run build`。
- [x] `admin` 首屏展示管理后台工作台，并包含公共功能、主要功能、机器人、邀请返利等模块入口。
- [x] `架构设计.md` 记录本轮工程结构和首期范围。
- [x] `TODO.md` 记录本轮关键开发步骤、问题和完成时间。

## 完成定义

- lint、typecheck、build 检查通过，或明确记录未能运行的原因。
- 新增代码遵循 `.trellis/spec/backend/` 与 `.trellis/spec/frontend/` 初始约定。
- 文档与实现保持一致。
- 后续未完成能力明确留在 `架构设计.md` 或 `TODO.md`。

## 暂不包含

- 手机端。
- 真实用户登录认证与权限鉴权。
- 数据库持久化、迁移和生产配置。
- 真实开奖 API 对接。
- 完整投注、计奖、派奖、合买满单、机器人自动执行逻辑。
- 充值支付、提现、客服实时通信。
