# 期号封盘投注校验

## 背景

当前订单创建只校验彩种、销售状态、玩法配置、余额和选号参数，`issue` 仍然只是一个字符串。这样会导致后台可以对不存在期号、已封盘期号、已开奖期号或已取消期号继续创建订单，后续计奖、财务和机器人流程都可能写入异常订单。

`架构设计.md` 的后续项已经明确要求：订单创建需要校验期号是否存在、是否仍在销售期，并在封盘后拒绝投注。

## 目标

1. 订单创建必须读取开奖期号，并校验期号和彩种匹配。
2. 只有状态为 `open` 的期号允许创建订单。
3. `closed`、`drawn`、`cancelled` 期号都必须拒绝投注。
4. 前端订单页面创建订单时优先从可投注期号中选择，避免手工输入不存在期号。
5. 文档、TODO 和 API 契约同步更新。

## 后端范围

1. 在订单创建路由中，根据 `lotteryId + issue` 查找开奖期号。
2. 新增开奖仓储查询能力，支持按彩种和期号查找。
3. 订单创建前校验：
   - 期号存在。
   - 期号彩种与订单彩种一致。
   - 期号状态为 `open`。
4. 订单结算仍按已有 draw issue ID 流程执行，不改变结算接口。
5. 增加单元测试覆盖 open 可下单、closed/drawn/cancelled 拒绝、期号不存在拒绝。

## 前端范围

1. 订单管理页面加载开奖期号列表。
2. 创建订单时，彩种选择后只展示该彩种 `open` 状态期号。
3. 期号不再优先手工输入；没有 open 期号时提示先创建期号。
4. 页面文案说明订单会校验期号销售状态。

## 非目标

1. 不实现真实时间比较封盘；本阶段以期号状态 `open/closed/drawn/cancelled` 作为销售状态来源。
2. 不实现定时自动封盘。
3. 不做开奖期号 PostgreSQL 持久化。
4. 不修改真实支付或提现能力。

## 验收标准

1. `POST /api/admin/orders` 对不存在期号返回 404 或业务错误信封。
2. `POST /api/admin/orders` 对 `closed/drawn/cancelled` 期号返回 400。
3. `POST /api/admin/orders` 对 `open` 期号正常创建订单并扣款。
4. 管理后台订单页面可从 open 期号下拉框选择期号创建订单。
5. `cargo fmt --check`、`cargo check`、`cargo test`、`npm run build` 通过。
