# 自动封盘开奖结算基础

## 目标

为开奖期号补齐基础自动执行链路：管理员可以在管理后台触发一次自动任务，后端按照当前时间扫描期号，自动关闭已到封盘时间的销售、自动开奖已到开奖时间且可自动开奖的期号，并在开奖后自动执行计奖派奖与资金入账。

## 已知信息

- 用户要求持续完善彩票管理后台，当前后续缺口集中在期号封盘、开奖、结算和资金闭环。
- `架构设计.md` 的后续需求明确写到：需要根据开奖时间实现自动创建期号、自动封盘、自动开奖任务，开奖后自动触发结算，并补齐失败重试。
- 当前后端已有期号仓储、封盘接口、开奖接口、结算接口和资金派奖入账能力，仍主要使用内存仓储。
- 当前订单创建已经校验期号必须存在且为 `open` 状态，自动封盘后应自然阻止继续投注。
- 当前开奖号码保存和接口返回统一使用英文逗号分隔格式，例如 `2,4,7`、`7,8,9,4,2`。

## 本阶段范围

- 新增后端自动任务执行能力，支持按传入的 `now` 时间字符串扫描现有期号。
- 自动关闭 `open` 状态且 `saleClosedAt <= now` 的期号。
- 自动开奖 `open` 或 `closed` 状态且 `scheduledAt <= now` 的期号。
- `platform` 和 `api` 开奖模式使用当前本地生成器自动生成开奖号码。
- `manual` 开奖模式缺少管理员指定开奖号码时不自动开奖，并在任务结果中记录跳过原因。
- 自动开奖成功后自动执行结算，并调用资金仓储做派奖入账。
- 新增管理后台入口，在“开奖期号与开奖源”页面展示自动任务执行区，可查看本次关闭、开奖、结算、入账和跳过结果。
- 更新 `架构设计.md` 和 `TODO.md`，记录本轮能力边界、解决的问题和验证结果。

## 不在本阶段范围

- 不实现独立后台常驻调度进程或系统级定时任务。
- 不实现自动创建下一期号。
- 不接入真实第三方开奖 API。
- 不实现失败重试队列、审计日志、撤销结算或重结算。
- 不把期号、订单、结算和资金流水迁移到 PostgreSQL 事务持久化。

## 验收标准

- [ ] 后端提供统一 API 信封的自动任务执行接口。
- [ ] 任务能把已到封盘时间的 `open` 期号变为 `closed`。
- [ ] 任务能把已到开奖时间的 `platform/api` 期号变为 `drawn`，开奖号码为英文逗号分隔格式。
- [ ] 自动开奖后能生成结算批次，并把中奖派奖写入资金流水。
- [ ] `manual` 期号不会被无号码自动开奖，返回结果包含跳过原因。
- [ ] 管理后台可以触发自动任务并刷新期号列表。
- [ ] `cargo fmt --check`、`cargo check`、`cargo test` 和 `npm run build` 通过。

## 技术备注

- 时间比较沿用当前期号字段的固定字符串格式 `YYYY-MM-DD HH:mm:ss`，本阶段使用传入的 `now` 字符串进行字典序比较，避免在内存阶段引入额外时间依赖。
- 自动任务执行器需要复用现有 `DrawRepository`、`OrderRepository` 和 `FinanceRepository`，避免绕过服务层状态校验。
- 重复结算仍由订单结算服务防护；本阶段自动任务只对本次新开奖成功的期号执行结算，减少重复调用风险。
