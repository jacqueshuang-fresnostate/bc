# 彩票系统完善首期：玩法规则引擎

## 目标

系统性补全彩票管理后台和后端业务能力。本轮先实现最底层的玩法规则引擎：根据 `3个号码玩法规则说明.md` 和 `5个玩法规则说明.md`，提供注数计算、投注展开、中奖判断和规则验证 API，让后续订单、计奖、派奖、机器人、财务和风控能建立在可测试的规则基础上。

## 已知信息

- 项目已完成 Rust 后端、React 管理后台、彩种管理 CRUD 和可选 PostgreSQL 彩种持久化。
- 管理后台仍有大量占位模块：用户、订单、财务、在线客服、管理员、角色权限、系统设置、开奖模式、开奖时间、合买配置、机器人、邀请、返利。
- `架构设计.md` 要求依据 3 位和 5 位玩法规则开设彩种。
- `3个号码玩法规则说明.md` 包含 3 位号码玩法：直选、组三复式、组三胆拖、组六复式、组六胆拖。
- `5个玩法规则说明.md` 包含 5 位号码的前三/中三/后三直选、直选组合、组三复式、组三胆拖、组六复式、组六胆拖、大小单双。
- 当前项目文档必须使用中文输出。
- 当前工作区存在用户或 IDE 产生的未提交变更：`admin/vite.config.ts`、`backend/src/main.rs`、`.idea/`。本任务会保留这些变更，不主动提交或覆盖。

## 整体系统补全路线

1. **玩法规则引擎**：注数计算、投注展开、中奖判断、规则验证 API 和后台验证页面。
2. **订单与投注**：创建投注订单、订单查询、订单状态、投注明细、金额校验。
3. **开奖源与期号**：平台随机开奖、API 开奖、指定号码、开奖时间、期号生成。
4. **计奖与派奖**：按订单和开奖结果结算中奖、生成派奖记录、更新余额。
5. **用户与资金**：用户管理、充值、提现、余额、流水、风控冻结。
6. **合买**：合买计划、份额、参与、满单、中奖分配。
7. **机器人**：合买机器人、购彩机器人、执行窗口、任务日志。
8. **权限与后台运营**：管理员、角色权限、系统设置、审计日志。
9. **邀请返利**：代理邀请、下级关系、立即返利、充值阶梯返利。
10. **手机端**：用户购彩、订单、资金、合买、邀请入口。

## 暂不包含

- 不创建真实投注订单。
- 不扣减用户余额。
- 不生成派奖记录。
- 不接入真实开奖源或期号表。
- 不实现奖金赔率、奖金分配或财务流水。
- 不实现手机端购彩。

## 数据契约草案

后端新增接口：

- `GET /api/admin/play-rules`
- `POST /api/admin/play-rules/evaluate`

请求体示例：

```json
{
  "numberType": "threeDigit",
  "ruleCode": "threeDirect",
  "selection": {
    "positions": [[2], [4], [7]]
  },
  "drawNumber": "247"
}
```

响应体示例：

```json
{
  "ruleCode": "threeDirect",
  "stakeCount": 1,
  "expandedBets": ["247"],
  "isWinning": true,
  "matchedBets": ["247"]
}
```

## 风险与约束

- 玩法规则是后续订单和派奖的核心，不能只靠前端计算。
- 当前没有赔率和奖金表，本轮只计算是否命中和命中投注。
- 5 位大小单双不同平台可能看不同位置，本轮采用规则文档里“通常看后两位”的默认口径，后续可配置。
- 用户已有端口和 IDE 配置变更，本任务提交时需要排除这些未识别变更。
