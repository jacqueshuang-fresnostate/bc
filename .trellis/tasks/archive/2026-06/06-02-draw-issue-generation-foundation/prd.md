# 自动创建下一期号基础

## 目标

为开奖期号补齐“按彩种开奖计划生成下一期”的基础能力。管理员可以在开奖期号页面选择彩种并触发生成下一期号，后端根据彩种的开奖时间配置计算下一次开奖时间、封盘时间和期号编码，从而让后续自动封盘、自动开奖和自动结算有稳定的期号来源。

## 已知信息

- 用户要求持续完善彩票管理后台，当前系统已经具备手动创建期号、封盘、开奖、结算和资金入账能力。
- 上一阶段已实现后台触发式自动封盘、自动开奖和自动结算，但仍需要管理员先手动创建期号。
- `架构设计.md` 的后续需求明确写到：需要自动创建下一期号，支持周期开奖、每日固定时间开奖和周开奖。
- 当前彩种模型已有 `DrawSchedule`：`periodic.intervalSeconds`、`daily.time`、`weekly.weekdays + time`。
- 当前仍以内存开奖仓储为主，本阶段不实现数据库事务、后台常驻调度或真实第三方开奖 API。

## 本阶段范围

- 新增后端“生成下一期号”请求与接口。
- 后端根据彩种开奖计划计算下一期：
  - 周期开奖：基于最新已存在期号开奖时间或传入基准时间，加上 `intervalSeconds`。
  - 每日固定开奖：选择严格晚于基准时间的下一次每日开奖时间。
  - 周开奖：选择严格晚于基准时间的下一次指定星期开奖时间。
- 后端自动生成期号编码，使用开奖时间格式化为 `YYYYMMDDHHMMSS`。
- 后端自动计算封盘时间，默认开奖前 30 秒，可由请求传入 `saleCloseLeadSeconds` 覆盖。
- 如果同彩种已有未来期号，则从该彩种最新开奖时间继续计算下一期，避免重复生成同一个计划点。
- 管理后台“开奖期号与开奖源”页面新增“按计划生成下一期”入口。
- 更新 `架构设计.md`、`TODO.md` 和 API 契约规格。

## 不在本阶段范围

- 不启动后台常驻调度进程。
- 不自动循环生成多期。
- 不接入真实第三方开奖 API。
- 不实现失败重试队列、任务审计日志或告警。
- 不把开奖期号迁移到 PostgreSQL。

## 验收标准

- [ ] 后端提供统一 API 信封的生成下一期接口。
- [ ] 周期开奖彩种能按 `intervalSeconds` 生成下一期。
- [ ] 每日固定开奖彩种能在当天或次日生成下一期。
- [ ] 周开奖彩种能按指定星期和时间生成下一期。
- [ ] 生成的期号、开奖时间、封盘时间能在开奖期号列表显示。
- [ ] 管理后台可以一键生成下一期并刷新列表。
- [ ] `cargo fmt --check`、`cargo check`、`cargo test` 和 `npm run build` 通过。

## 技术备注

- 时间字段继续沿用 `YYYY-MM-DD HH:mm:ss` 字符串格式，并在后端服务层做解析和格式化。
- 本阶段可以引入轻量时间库处理日期加减和星期计算，避免手写闰年、月份天数和星期算法。
- 生成接口仍复用现有 `DrawRepository::create`，避免绕过期号创建校验和重复期号保护。
