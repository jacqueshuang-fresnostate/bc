# 代码复用思考指南

> **用途**：写新代码前先判断是否已有相似逻辑。

---

## 问题背景

重复代码是行为不一致的主要来源。

复制或重写已有逻辑会带来：

- 缺陷修复无法同步到所有副本。
- 行为随时间逐渐分叉。
- 代码库越来越难理解。

---

## 写新代码前

### 1. 先搜索

```bash
rg "functionName" .
rg "keyword" .
```

### 2. 问自己这些问题

| 问题 | 如果答案是是 |
|------|--------------|
| 是否已经有类似函数？ | 使用或扩展已有函数 |
| 是否已经有类似模式？ | 遵循现有模式 |
| 是否应该成为共享工具？ | 放到正确的共享位置 |
| 是否正在从另一个文件复制代码？ | 停下来，考虑抽取共享逻辑 |

---

## 常见重复模式

### 模式 1：复制函数

不好的做法：把校验函数复制到另一个文件。

好的做法：抽到共享工具中并按需导入。

### 模式 2：相似组件

不好的做法：创建一个和现有组件 80% 相似的新组件。

好的做法：通过 props 或小范围扩展复用已有组件。

### 模式 3：重复常量

不好的做法：在多个文件定义同一常量。

好的做法：建立单一事实来源并在各处导入。

---

## 何时抽象

应该抽象：

- 相同代码出现 3 次以上。
- 逻辑复杂到容易出错。
- 多个开发者或多个模块会使用。

不应抽象：

- 只使用一次。
- 只是简单一行。
- 抽象本身比重复更复杂。

---

## 批量修改后

当你在多个文件做了相似修改：

1. 回看是否覆盖了所有实例。
2. 用 `rg` 搜索是否有遗漏。
3. 判断是否应该抽成共享模块或组件。

---

## 本项目提示

- 后端 API 响应统一使用 `ApiEnvelope`，不要在各路由重复定义响应形状。
- 前端 API 调用集中在 `src/api/`，不要在页面组件里重复 `fetch`。
- 管理后台重复面板优先抽成共享组件，例如指标卡和模块面板。
- 彩票玩法、开奖模式、返利模式等枚举需要保持后端和前端命名一致。
