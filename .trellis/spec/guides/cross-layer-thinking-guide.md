# 跨层思考指南

> **用途**：实现前先想清楚数据如何跨层流动。

---

## 问题背景

很多缺陷发生在层边界，而不是某一层内部。

常见跨层问题：

- API 返回格式 A，前端却期望格式 B。
- 数据库存储 X，服务层转换成 Y 时丢失字段。
- 多个层分别实现同一段逻辑，后续行为逐渐不一致。

---

## 实现跨层功能前

### 1. 画出数据流

```text
来源 -> 转换 -> 存储 -> 读取 -> 转换 -> 展示
```

每个箭头都要问：

- 数据当前是什么格式？
- 可能在哪里出错？
- 哪一层负责校验？

### 2. 识别边界

| 边界 | 常见问题 |
|------|----------|
| API ↔ Service | 类型不匹配、字段缺失 |
| Service ↔ Database | 格式转换、空值处理 |
| Backend ↔ Frontend | 序列化、日期格式、字段命名 |
| Component ↔ Component | props 形状变化 |

### 3. 定义契约

每个边界都要明确：

- 输入格式是什么？
- 输出格式是什么？
- 可能返回哪些错误？

---

## 常见跨层错误

### 错误 1：隐式格式假设

不好的做法：没有确认日期格式就直接使用。

好的做法：在边界处显式转换格式。

### 错误 2：校验分散

不好的做法：多个层重复校验同一件事。

好的做法：在入口处完成校验，并让下游依赖明确契约。

### 错误 3：抽象泄露

不好的做法：组件知道数据库表结构。

好的做法：每层只知道相邻层的契约。

## 跨层功能检查清单

实现前：

- [ ] 已画出完整数据流。
- [ ] 已识别所有层边界。
- [ ] 已定义每个边界的数据格式。
- [ ] 已决定校验发生在哪一层。

实现后：

- [ ] 已用空值、空列表、无效数据等边界情况检查。
- [ ] 已验证每个边界的错误处理。
- [ ] 已确认数据能从后端到前端完整显示。

---

## 本项目提示

彩票管理后台首期同时涉及后端 API、服务层、领域类型、前端 API client、hook 和页面组件。实现时优先保持以下契约一致：

- 后端 JSON 字段使用 `camelCase`。
- 前端类型必须与后端响应字段一一对应。
- 金额字段使用 `amountMinor`、`balanceMinor` 等最小货币单位字段，不在前端假设浮点金额。
- 演示数据和真实数据路径要明确区分。

## 何时编写详细流程文档

遇到以下情况时，需要创建详细流程文档：

- 功能跨越 3 层以上。
- 涉及多个团队或多个开发阶段。
- 数据格式复杂。
- 同类功能曾经导致缺陷。
