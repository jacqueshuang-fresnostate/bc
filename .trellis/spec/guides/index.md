# 思考指南

> **用途**：在编码前扩展思考范围，提前发现容易遗漏的问题。

---

## 为什么需要思考指南

大多数缺陷和技术债不是能力不足，而是“当时没想到”：

- 没想清楚层与层之间的数据格式，容易产生跨层缺陷。
- 没意识到重复模式，导致到处复制代码。
- 没覆盖边界情况，导致运行时错误。
- 没考虑后续维护者，导致代码难读难改。

这些指南帮助你在编码前问对问题。

## 快速触发条件

### 需要思考跨层问题时

- [ ] 功能触及 3 层以上，例如 API、Service、Component、Database。
- [ ] 数据格式会在层之间变化。
- [ ] 多个消费者需要同一份数据。
- [ ] 不确定某段逻辑应该放在哪一层。

阅读 [跨层思考指南](./cross-layer-thinking-guide.md)。

### 需要思考代码复用时

- [ ] 正在写类似已有代码的逻辑。
- [ ] 同一模式重复出现 3 次以上。
- [ ] 新增字段需要同步多个地方。
- [ ] 正在修改任何常量或配置。
- [ ] 正在创建新的工具函数。

阅读 [代码复用思考指南](./code-reuse-thinking-guide.md)。

---

## 修改前搜索规则

> 修改任何值之前，先搜索相关引用。

```bash
rg "value_to_change" .
```

这个习惯能避免很多“漏改一个地方”的问题。

---

## 使用方式

1. 编码前阅读相关指南。
2. 编码中发现重复或复杂边界时回到指南检查。
3. 修复缺陷后，把值得沉淀的经验补进对应指南。

---

## 贡献方式

发现新的“当时没想到”的问题时，把它补充到对应指南。

---

**核心原则**：提前 30 分钟想清楚，能节省后面数小时排错。
