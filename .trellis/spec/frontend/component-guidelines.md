# 组件规范

> 本项目组件构建方式。

---

## 概览

组件应呈现运营控制台的感觉：简洁、可扫描、可预期。导航、卡片、表格、按钮、标签、提示和加载状态优先使用 Semi UI。布局和间距使用 Tailwind。

---

## 组件结构

- 使用命名导出。
- props 接口放在拥有该组件的文件附近，除非多个文件共享。
- 组件职责保持聚焦；页面重复同一模式时再抽取共享展示控件。
- 避免多层卡片嵌套。

示例：

```tsx
interface MetricCardProps {
  label: string;
  value: string;
}

export function MetricCard({ label, value }: MetricCardProps) {
  return <Card>{value}</Card>;
}
```

---

## Props 约定

- 优先使用明确的 prop 名称，避免大型无类型配置对象。
- 只有明确需要插槽时才使用 `ReactNode`。
- 避免用大量可选 props 制造很多视觉模式；行为分叉明显时拆组件。
- 回调 props 使用 `on` 前缀，例如 `onRefresh`。

---

## 样式模式

- Tailwind 用于布局、网格、间距和响应式。
- 能用 Semi UI 组件 prop 表达的变体，优先不用自定义 CSS。
- 页面背景保持克制并服务后台工作。
- 卡片只用于真实分组数据或重复项，不用于装饰页面区块。
- 后台列表页的创建/编辑维护表单如不需要常驻对照，应使用 Semi UI `SideSheet` 打开；主页面保留列表、筛选、统计和“新建/编辑”入口，避免右侧表单长期占用列表扫描空间。
- `SideSheet` 表单保存、删除成功后应关闭抽屉，并沿用页面原有 hook 或 API 刷新链路；切换模块时应关闭已打开的维护抽屉，防止不同模块的编辑状态残留。
- 后台彩种新增/编辑表单的开奖时间类型必须和后端 `DrawSchedule` 保持一致；当前支持“周期开奖、时间节点周期、每日固定、周开奖”。新增或修改排期类型时，彩种列表、dashboard、彩种控制台和手机端聚合文案都要同步更新。
- 后台订单、财务等运营列表如果支持“显示机器人数据”开关，默认必须关闭；开关文案使用“显示机器人数据”，打开后才把 `includeRobotData=true` 传给后端。
- 后台机器人配置页面需要按后端返回的 `deletable` 控制删除入口：普通机器人配置可删除，核心内置机器人删除按钮必须禁用并提示“内置机器人不能删除”，仍只能通过状态切换暂停或禁用。前端不能自行硬编码受保护机器人 ID。
- 后台机器人配置页面只展示“合买机器人”和“补单机器人”两类职责：合买机器人只负责发起合买计划到合买大厅，列表策略列显示“只发合买”，表单不展示补单策略；补单机器人只负责认购未满单合买，表单才展示“阶段性补单 / 开奖前补满”、阶段单次最高百分比和开奖前秒数配置。页面文案不要再使用“购彩机器人”。
- 后台机器人配置页面需要展示“机器人独立调度”状态卡，使用 `/api/admin/robot-scheduler/status` 和 `/api/admin/robot-scheduler/config` 维护常规机器人自动执行开关和周期；不要把常规机器人启停放回开奖调度页面，开奖调度页面只负责封盘、开奖、结算和流单前兜底。
- 客服会话消息流应使用 Semi UI `Chat` 组件承载，不手写消息气泡列表；后台客服“消息记录”按客服处理视角展示，用户消息在左侧、客服回复在右侧，头像文字和颜色必须依据真实消息作者而不是仅依据 `Chat` 角色显示。当后台页面只需要展示历史消息、回复输入由业务表单承担时，需要设置 `renderInputArea={() => null}` 和 `enableUpload={false}`，避免出现重复输入区或默认上传控件警告。后台客服回复框按 `Enter` 发送，`Shift+Enter` 等组合键保留换行，且必须检查输入法组合态避免中文选词回车误发。后台客服回复区如接入表情，使用 `@emoji-mart/react` 与 `@emoji-mart/data`，并通过动态 `import()` 懒加载选择器、数据和中文语言包；选中后插入到当前 `textarea` 光标位置，不能把完整表情数据静态打进后台首屏主包。表情面板由 Semi UI `Popover` 负责显示与外部点击关闭，`Popover` 需要设置 `keepDOM` 复用 `emoji-mart` Picker 实例，并通过 `onClickOutSide` 关闭；不要同时给 Picker 传 `onClickOutside`，避免 Picker 内部监听在弹窗关闭后残留，导致第二次点击“表情”刚打开就被关掉。后台客服图片回复必须复用图床上传接口，消息按 `messageType=image` 和 `imageUrl` 渲染缩略图；后台在线客服列表不得展示 `status=closed` 的已关闭会话，状态筛选使用 Tabs 区分“全部、处理中、等待用户、已解决”，不要只靠颜色标签让客服人工扫描。后台在线客服列表和详情必须展示用户上级代理，使用后端 `agentId/agentUsername`；没有上级代理显示“无”，只有代理 ID 但用户名缺失时显示“未知代理”，实时客服事件缺少代理字段时必须保留当前列表已有代理显示。后台在线客服主页面不保留顶部 `grid gap-3 sm:grid-cols-2 xl:grid-cols-4` 统计卡片区，标题和错误提示后直接进入会话列表与详情。在线客服页面内刷新只刷新客服会话数据，不能触发全局模块刷新导致浏览器刷新或局部刷新后跳回其它后台页面。客服列表排序必须未读优先，再按最后消息或更新时间倒序；未读会话必须使用 Semi UI `Badge` 标记未读数量，并用轻量底色或红点辅助扫描。客服仍可把当前会话保存为已关闭，保存后该会话应从列表中移除；会话状态为“已解决”时详情区显示“删除会话”按钮，点击后调用删除接口并从列表移除，未解决会话不能露出删除入口。
- 手机端彩票卡片展示开奖号码时，不能按卡片变体写死 3 位；必须优先使用真实 `latestResult.length`，再结合后端 `resultCount` 决定展示数量，兼容 3 位和 5 位彩种。没有开奖结果时才按该数量用期号尾号或 `?` 补位。
- 手机端首页“高频极速”推荐区的开奖号码必须使用固定正圆号码球，样式需要同时约束 `width`、`height`、`aspect-ratio: 1 / 1` 和 `border-radius: 9999px`，不要只依赖文字内容或内边距撑开形状。
- 手机端首页“高频极速”推荐区默认由后台 `mobile_home_featured_enabled` 关闭；开启后只展示 `mobile_home_featured_lottery_codes` 配置的销售中彩种，不能按开奖周期自动兜底展示。
- 手机端首页“高频极速”二级彩种卡片必须保持可扫描：状态标签、倒计时和值/入口标签都要 `white-space: nowrap`，不能因为两列宽度不足把“可下注”挤成逐字竖排；二级卡优先整卡点击加小入口标签，不使用占满底部的大按钮撑高卡片。
- 手机端首页所有彩种卡片都不展示合买标签、合买按钮或合买大厅入口；合买功能只在合买大厅、下注页合买模式和我的合买等专用页面展示。
- 手机端首页顶部可以使用轻量淡红色线性渐变氛围层和半透明 Header 提升质感，但不能使用强装饰形状、大面积高饱和红色或影响 Banner、钱包入口和彩种卡片点击的覆盖层；氛围层必须 `pointer-events: none`。
- 手机端首页普通分类彩种卡片使用顶部名称/状态/期号与右侧彩种 logo、底部整宽开奖号码的纵向结构；状态标签和期号必须分行展示，避免状态标签挤占期号空间。开奖号码行必须脱离左侧文案区域并放在卡片底部，单行不换行，号码位数较多时允许号码行内部横向滑动，不要让号码换行撑高卡片。不要再用头像和号码上下堆叠撑高卡片。logo 容器需要固定宽高和圆角，缺失 logo 时显示同尺寸 fallback，保证两列布局高度稳定。首页彩种卡片整体优先服务批量扫描，普通卡片应保持紧凑 padding、较小 logo 和小号号码球，避免一屏只能展示少量彩种。
- 手机端首页普通彩种分类需要使用 `van-tabs` 做分类切换，每个分类作为一个 Tab，只渲染当前分类下的两列彩种卡片；不要把所有分类纵向全部铺开，避免首页过长、同屏信息密度下降。
- 手机端首页彩种卡片倒计时展示口径为“距开奖时间”，应优先使用后端 `nextDrawTime` / 实时开盘事件里的 `scheduledAt`；`saleStopTime` 只用于状态和投注禁用链路，不作为首页主倒计时。倒计时到达开奖时间后必须触发静默刷新，且需要消费 `issue_opened` / `issue_closed` 实时事件更新当前期号和封盘状态，避免长期停留在“开奖中”。
- 手机端开奖历史和注单状态展示必须通过中文状态映射处理，不能把 `drawn`、`pendingDraw` 等接口状态值直接渲染给用户。
- 手机端“开奖结果”列表和单彩种全部开奖弹层的号码球必须保持紧凑，默认直径控制在 32px 左右，小屏控制在 30px 左右；不要复用首页主推大卡的大号码球尺寸，避免开奖列表被号码区域撑高。
- 手机端充值页必须以 `GET /api/user/recharge/config` 返回的后台充值配置为准，只展示已开启且真正可用的 `rainbowEpay` 和 `customerService` 渠道；彩虹易支付必须使用后端 `payTypes`，当 `payTypes` 为空时不能展示在线支付入口，也不能默认补 `alipay`。客服直充创建订单后跳转绑定的客服会话。不要再调用旧 `/payment/*` 接口，也不要在后端未配置时展示 USDT 或快捷充值模式。
- 手机端充值页属于高频资金操作页，需要直接展示充值渠道卡片、余额摘要、快捷金额和底部固定提交栏；快捷金额必须按后台 `minAmountMinor/maxAmountMinor` 过滤，最近订单中可继续处理的彩虹易支付订单应提供“继续支付”，客服直充订单应提供“联系客服”。
- 手机端充值页如果后端返回 `bonusEnabled=true` 且 `bonusRules` 非空，必须展示充值赠送活动档位，并按当前输入金额提示预计赠送；展示金额使用元，接口契约继续使用分。后台和手机端资金流水都必须把 `rechargeBonusCredit` 映射为“充值赠送”，不要混同为充值本金或代理返利。
- 手机端充值页金额输入属于浏览器原生 `type="number"` 输入，运行时可能返回字符串或数字；金额解析函数不能直接调用 `value.trim()`，必须先用 `String(value ?? '').trim()` 归一化，再按两位小数转换为最小货币单位。
- 手机端资金类金额输入不能在每个按键后立即强制格式化或夹到最小金额，否则用户按删除键、选择文本或粘贴时会被立刻回填；应允许编辑态为空或部分小数，失焦、回车、快捷金额、加减按钮或提交时再做最终归一化。
- 手机端登录/注册页必须在一个视口内完整展示，不允许依赖页面滚动；根容器优先基于 `html/body/#app` 的 `height: 100%` 使用满高布局，避免 `100dvh` 在部分 WebView 或浏览器环境里比实际可视高度多出少量像素。品牌区、Logo、表单卡片、输入框、按钮和切换入口都要使用紧凑尺寸，小高度屏幕继续收紧间距，并且不要在登录页保留会撑高首屏的页脚。
- 手机端注册页不要用浏览器语言、系统地区或时区推断注册地，也不要把这些字段作为 `registrationLocation` 上报；注册提交时 Android/iOS Tauri App 优先通过 `@tauri-apps/plugin-geolocation` 获取经纬度，H5 通过浏览器 `navigator.geolocation` 获取经纬度，并统一以 `registrationPosition` 上报给后端。定位失败、用户拒绝授权或 H5 非 HTTPS 不阻断注册，由后端从代理请求头记录注册 IP 兜底；手机端不得自行拼接国家、省市名称。
- 手机端固定或吸顶顶部栏必须接入公共安全区样式：品牌型 Header 使用 `mobile-safe-header`，紧凑返回栏使用 `mobile-safe-compact-header`，固定 Header 后面的主内容使用 `mobile-safe-main-top`、`mobile-safe-main-top-tight` 或 `mobile-safe-main-top-loose`。不要直接写裸 `fixed top-0 h-16` 或 `sticky top-0 h-14`，否则 Tauri Android 真机会让 Header 被状态栏遮挡。聊天/客服等 scoped CSS 顶部栏也必须使用 `--mobile-status-safe-top` 变量同步留白。
- 手机端所有页面 Header 背景和底部主导航背景必须和首页保持一致，统一使用 `--mobile-app-header-background`、`--mobile-app-header-border` 和 `--mobile-app-header-shadow`；能用 `mobile-safe-header` / `mobile-safe-compact-header` 的页面不要再单独写白底 Header，自定义聊天顶部栏和 `mobile-bottom-nav` 里的导航容器也要复用这组变量。底部主导航、聊天大厅输入区和依赖底部导航的页面留白必须使用 `--mobile-bottom-nav-safe-bottom`、`--mobile-bottom-nav-bottom-offset`、`--mobile-bottom-nav-extra-lift` 或 `--mobile-bottom-nav-space`，不要直接只写 `env(safe-area-inset-bottom)`；安卓三键虚拟导航机型需要通过 `visualViewport` 兜底抬起底栏，真机微调底栏高度时优先调整 `--mobile-bottom-nav-extra-lift`。
- 手机端老安卓兼容要求：余额卡、设置列表、充值/提现按钮、首页彩种卡片、下注底栏等关键入口不能只依赖 Tailwind 的渐变变量、透明色、任意阴影或复杂工具类组合；组件应保留语义类并补普通 scoped CSS 兜底，例如直接写 `background: linear-gradient(...)`、`border-radius`、`box-shadow`、按钮文字色和图标色，避免旧 Android WebView 局部忽略工具类后出现白字落浅底、卡片背景消失或按钮颜色异常。
- 手机端页面不允许用户通过拖动、双指或双击把整个 WebView 页面放大；根节点必须保持 `viewport user-scalable=no`、`touch-action: pan-x pan-y` 和 `installMobileTouchZoomGuard()` 三层防护。新增页面、弹窗或滚动容器不得覆盖根级 `touch-action` 为 `auto` 或 `pinch-zoom`，只处理自身滚动和点击反馈。触摸防护只能拦截双指缩放和短距离双击，必须通过触摸位移区分普通上下滚动，不能在滚动 `touchend` 上直接 `preventDefault()`。
- 手机端聊天、客服、评论等固定底部输入区必须消费 `--mobile-keyboard-bottom-inset`，让输入法弹起时输入栏、表情面板、新消息提示和消息列表底部留白同步上移；不要只写 `bottom: 0` 或只依赖 `env(safe-area-inset-bottom)`。如果页面同时有底部主导航，输入栏 bottom 需要在底部导航空间和键盘高度之间取较大值。
- 手机端所有显示钱包余额的 Header 必须使用公共 `WalletHeaderAmount` 组件，并通过 `walletPrivacy` Pinia store 共享“隐藏/显示金额”本地偏好；隐藏时展示 `••••` 和可切换图标，不能在各页面直接写死 `¥{{ balance }}`。
- 手机端弹窗要按用途限制高度，避免默认做成接近整页：普通选择弹层建议控制在 `62-66dvh`，详情和购物篮控制在 `66-68dvh`，表单/创建类控制在 `70dvh` 左右，聊天红包/合买选择控制在 `64dvh` 左右。内容较长时必须让弹窗内部滚动，不要把外层高度放到 `85-90vh`。`emoji-mart` 表情面板必须显式设置 `dynamicWidth`、较小 `emojiButtonSize/emojiSize` 和约 `300px` 高度，避免默认 `435px` 面板在真机上过大。
- 手机端所有展示给用户的错误提示都必须走 `mobile/src/utils/errorMessage.ts` 的 `errorMessage` 或 `userFacingErrorMessage`；不得直接把后端 `message/detail`、Axios `err.message`、`Network Error` 或 `Request failed with status code ...` 原样传给 `showToast/showNotify`，避免出现英文错误提示。
- 手机端“我的账户”资金流水必须使用当前系统的 `GET /api/user/ledger-entries` 当前用户接口，只展示登录用户自己的流水；页面不得调用后台全量资金流水接口，也不需要为旧系统字段做兼容。流水条目不展示后端 `referenceId` 关联单号，避免把内部关联编号暴露给普通用户。
- 手机端“我的账户”头像设置必须使用用户端 `POST /api/user/avatar/upload` 或 `PUT /api/user/avatar`，不能复用后台 `/api/admin/image-bed/upload`；上传成功后需要同步刷新页面资料和 Pinia 登录态，保证返回个人中心或重新打开应用时继续展示最新 `avatarUrl`。头像点击上传优先使用原生 `input[type="file"]` 与 `label for` 绑定，避免自定义上传插槽在移动端点击不触发；头像视觉必须同时固定宽高并使用 `border-radius: 9999px` 保持正圆。
- 手机端头像图片展示必须通过公共 `CachedAvatarImage` 和 `avatarImageCache` 缓存，优先读取内存和本地 data URL 缓存；Tauri 打包场景可通过 Rust 命令下载远程头像并转为 data URL，避免个人中心和聊天大厅对同一个图床头像地址反复请求。缓存失败时才能回退原始 URL。
- 手机端通过 Tauri 打包 APK 时，`mobile/src-tauri/capabilities` 必须显式声明主窗口需要的能力；凡是前端使用 `@tauri-apps/plugin-store`、剪贴板插件或自定义 `invoke` 命令，都要同步维护 capability，并在 Android 构建后检查目标目录生成的 `capabilities.json` 不为空。

- 后台开奖源维护表单的供应商下拉必须覆盖后端 `DrawSourceProvider` 的全部枚举：`api68`、`kjApi`、`bbKaijiang`、`indonesiaLottery`；`bbKaijiang` 的编码字段用于 `gameCodeList`，`indonesiaLottery` 的编码字段仅作后台展示，展示文案可写为 `lotCode / lotKey / gameCode`。
- 后台彩种控制台整体采用“顶部可控销售彩种 Tab + 当前彩种详情面板”的布局；详情面板必须同时展示相关用户下单信息和合买发起记录，默认聚焦当前期或当前控制范围。控制范围必须用 Semi UI `Select` 明确区分“整个彩种”“指定期号”“指定订单所在期号”，订单范围只能选择待开奖订单，并把订单期号同步为控制期号。“控制期号”下拉只展示未结束期号，不能把 `drawn` 已开奖或 `cancelled` 已取消期号作为可选项。控制范围为“指定期号”且所选期号已经开奖或已取消时，前端必须自动取消“启用控制开奖”勾选并提示，后端保存接口也必须兜底关闭该控制，避免已结束期号残留控奖配置。
- 后台订单管理和彩种控制台控单详情面板必须直接展示用户下注信息：`selection` 要翻译成中文位置、选号、胆码拖码或大小单双。订单管理列表以扫描为主，只展示 `selection` 翻译结果，不展示展开注码 Tag；彩种控制台控单、订单详情或其它需要控单的明细场景才展示 `expandedBets` 展开注码标签。订单列表必须独立显示期号，并在用户列展示用户名和用户 ID。
- 后台资金流水、订单、计奖派奖、彩种控制台控单、充值、提现、客服会话等需要按用户核对的运营表格，用户列必须同时展示用户名和用户 ID；用户名缺失时展示“未知用户”，但用户 ID 仍必须保留，避免财务和控单排查断链。
- 后台财务管理、订单管理和手机端我的账户展示资金流水、充值、提现、订单创建/结算等业务时间时，必须通过统一时间格式化函数处理；标准展示为 `YYYY-MM-DD HH:mm:ss`，历史 `unix:秒` 标签和秒级时间戳需要转为北京时间，不得原样展示 `unix:...`。
- 后台订单管理的“创建投注单”属于维护表单，必须通过顶部按钮打开 Semi UI `SideSheet`；订单列表页面不保留右侧常驻创建表单，创建成功后关闭抽屉并刷新列表。
- 后台彩种管理列表必须按销售状态提供 Semi UI `Tabs` 分类，包括“全部、销售中、已停售”；列表数据持续增长时必须复用公共 `PageControls` 分页。销售状态列使用 Semi UI `Switch` 控制，旁边保留中文状态标签，不能退回普通按钮或只用颜色表达状态。
- 后台彩种新增/编辑表单必须提供“开奖号码控制”开关，并提交 `drawControlEnabled`；彩种控制台只能在该开关开启时展示“控制”按钮，关闭时不得让运营误以为该彩种可以控开。
- 后台彩种“避奖策略”属于彩种控制台操作，不属于彩种管理基础资料维护；彩种管理列表和新增/编辑 SideSheet 不展示避奖开关，彩种控制台当前彩种详情面板通过 Semi UI `Switch` 切换 `avoidWinningEnabled`。
- 后台彩种新增/编辑表单在开奖模式选择“API 接口”时必须显示“API开奖延迟秒数”输入项；该值提交为非负整数，只用于调度延迟请求开奖源，不能用来改变前端官方开奖倒计时。
- 后台彩种新增/编辑表单必须显示“封盘提前（秒）”输入项并提交 `saleCloseLeadSeconds`；最小值为 `1`，说明文案要明确该值表示“开奖前提前多少秒封盘”，例如 300 秒周期配置 60 秒时，在周期进行到 240 秒封盘并等待开奖，到开奖点才开启下一期。
- 后台彩种新增/编辑表单在开奖模式选择“平台开奖”时必须显示“平台期号格式”输入项；默认 `{date}{seq4}`，展示说明必须提示会生成类似 `202606130001` 的日期加 4 位序号，并列出 `{seq1}`、`{seq2}`、`{seq3}`、`{seq4}` 四种每日递增序号变量；提交字段为 `issueFormat`，只影响平台开奖期号生成，不能影响 API 开奖源期号同步。
- 后台开奖管理的自动任务结果和调度历史只展示跳过数量汇总，不展开 `skippedIssues`、`skippedLotteries` 的逐条黄色列表；停售彩种或旧期号跳过属于常规调度结果，避免把页面刷成告警墙。真实错误仍使用红色错误提示展示。
- 后台合买管理列表必须在最右侧提供“操作”列，点击“查看详情”后才加载并通过 Semi UI `SideSheet` 显示对应计划维护区；列表必须展示每条计划的创建时间，方便运营核对计划生成顺序。页面初次加载和创建计划后不能自动展开“参与记录”或“计划详情”。详情抽屉必须先展示“计划详情”，再展示“参与记录”；主页面不保留常驻详情卡片、顶部统计卡片或右侧选择提示，避免挤占列表扫描空间。
- 后台合买管理的“新增合买计划”属于维护表单，必须通过顶部按钮打开 Semi UI `SideSheet`；合买列表页面不保留常驻新增表单，创建成功后关闭抽屉并刷新列表，不能自动打开刚创建计划的详情。
- 后台合买管理计划列表需要提供“显示机器人数据”开关，默认关闭并只展示非机器人发起计划；开关打开时传 `includeRobotData=true`，并把分页重置到第 1 页。
- 后台合买管理需要提供“一键清除合买计划列表”按钮；按钮在加载中、保存中或当前口径无记录时禁用，点击后必须用中文确认清理范围和风险。清理调用 `DELETE /api/admin/group-buy/plans/clear`，成功后关闭详情抽屉、回到第 1 页、刷新列表和 dashboard；后端只删除已取消或已结算计划，未结算计划自动保留，前端成功提示需要说明“未结算计划已保留”。
- 后台财务管理资金流水标签页需要提供“一键清除”按钮；按钮在加载中、保存中或没有流水时禁用，点击后必须用中文确认“只清除流水审计列表，不会回滚或调整用户余额，后续新流水编号不会重置”。清理调用 `DELETE /api/admin/ledger-entries/clear`，成功后回到第 1 页并刷新 dashboard。
- 后台财务管理资金流水标签页需要提供类型筛选下拉；默认“全部类型”不传 `kind`，选择具体类型时传 `LedgerEntryKind` camelCase 枚举值并回到第 1 页。类型筛选必须走 `GET /api/admin/ledger-entries` 的仓储分页查询，不得在前端读取全量流水后筛选。
- 后台用户权限管理主页面不展示顶部统计卡片；用户管理、管理员管理和角色权限应优先展示模块切换、分页列表、排序和 SideSheet 维护入口，避免 `grid gap-3 sm:grid-cols-2 xl:grid-cols-4` 统计区挤占列表扫描空间。
- 后台侧边栏中 `registration` 模块展示名使用“系统配置”，用于维护用户名注册、邮箱注册和邀请码要求；不要再显示为“用户注册”，避免和用户端注册入口混淆。
- 后台侧边栏模块按“常用 / 不常用”分组展示时，分组标题必须支持折叠；“常用”默认展开，“不常用”默认收起。刷新恢复到不常用模块时，应自动展开对应分组，避免当前页面入口被隐藏。
- 后台用户、订单、财务、合买、计奖派奖等运营列表只要数据会持续增长，就必须使用分页结构和公共 `PageControls`，页面顶部显示总数、每页条数、上一页和下一页；用户维护等需要排序的列表必须通过后端白名单排序字段查询，用户维护列表默认使用降序，切换排序规则或“显示机器人数据”等影响列表口径的开关时应回到第 1 页。
- 后台所有原生 `<table>` 需要保持标准单层 `<thead><tr><th>` 结构，以便 `useResizableAdminTables` 自动添加表头右侧拖拽热区；新增表格不要手写独立列宽拖拽逻辑，也不要移除全局 `admin-resizable-table` 行为。
- 后台系统设置中的外部通知配置按 `support_telegram_` 前缀归入“通知设置”标签页；开启/关闭类配置使用 Semi UI `Select`，Token、Chat ID 等敏感配置只作为系统设置值维护，不硬编码到前端代码。
- 后台系统设置里的维护类动作（例如“刷新内存缓存”）必须和普通页面刷新区分：按钮需要中文确认提示，执行中显示 loading，成功后展示已处理模块摘要并重新拉取后台服务端数据；失败时使用中文 Toast，不要只静默刷新页面。

错误示例：

```vue
<div v-for="digit in digits(3)">{{ digit }}</div>
```

正确示例：

```ts
const displayCount = Math.max(lottery.latestResult?.length || 0, lottery.resultCount || 0, 3)
const displayDigits = roundDigits(lottery, displayCount)
```

> **注意**：当前 `@douyinfe/semi-ui` 包的 `exports` 不暴露 `dist/css/semi.min.css` 作为 bare import。Vite 构建中需要通过相对路径导入完整样式：
>
> ```ts
> import '../node_modules/@douyinfe/semi-ui/dist/css/semi.min.css';
> ```
>
> 如果升级 Semi UI 后官方暴露了新的样式入口，需要先更新本规范，再调整代码。

## 可访问性

- 图标按钮需要清晰标签或 tooltip。
- 加载和错误状态必须可见。
- 不要只依赖颜色表达状态。
- 文本在移动端和桌面端都需要可读且不溢出。

---

## 常见错误

- 不要为管理后台创建营销型 hero 页面。
- 不要在仪表盘和面板中使用过大的字体。
- 不要让接口失败时只显示空屏。
