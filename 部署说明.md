# 部署说明

## 本地联调

普通功能验证不需要打包 Docker 镜像，直接启动本地后端和前端服务。

### 后端环境变量

后端会在启动时读取项目根目录和 `backend/` 下的 `.env`、`.env.local`。真实本地配置写入 `.env.local`，该文件已被 `.gitignore` 忽略；可提交的示例配置在 `.env.example`。

当前本地 PostgreSQL 连接模板：

```bash
DATABASE_URL=postgres://root:<密码>@192.168.2.3:15432/postgres
PORT=18120
DATABASE_MAX_CONNECTIONS=30
DATABASE_MIN_CONNECTIONS=2
DATABASE_ACQUIRE_TIMEOUT_SECONDS=10
REDIS_URL=
```

并发接口压测或真实部署时，需要根据 PostgreSQL 的 `max_connections` 和机器规格调整连接池：

- `DATABASE_MAX_CONNECTIONS`：后端业务数据库最大连接数，默认 `30`。并发接口容易排队时优先提高它，但不要超过数据库可承受连接数。
- `DATABASE_MIN_CONNECTIONS`：启动后保留的最小连接数，默认 `2`。
- `DATABASE_ACQUIRE_TIMEOUT_SECONDS`：等待可用数据库连接的最长秒数，默认 `10`，超过后接口返回中文内部错误。
- `REDIS_URL`：可选 Redis 地址；留空时禁用，例如需要启用时再填写 `redis://127.0.0.1:6379`。配置后普通下注会先按用户获取短时分布式锁，同一用户下注串行，不同用户继续并发。

启动后端：

```bash
cd backend
cargo run
```

健康检查：

```bash
curl http://127.0.0.1:18120/api/health
```

### 前端环境变量

管理后台使用 `admin/.env.local` 配置后端地址，示例文件为 `admin/.env.example`：

```bash
VITE_API_BASE_URL=http://127.0.0.1:18120
```

启动前端：

```bash
cd admin
npm run dev -- --host 127.0.0.1 --port 5173
```

访问地址：

- 管理后台：`http://127.0.0.1:5173/`
- 后端 API：`http://127.0.0.1:18120/api/health`

## Docker 单镜像

本项目的 Docker 镜像把管理后台和后端 API 打包在同一个容器中：

1. `admin/` 使用 Node 构建为静态资源。
2. `backend/` 使用 Rust 构建 release 二进制。
3. 运行时使用 Nginx 服务前端页面，并把 `/api/` 反向代理到同容器内的后端服务。

### 构建镜像

```bash
docker build -t bc-platform:local .
```

### 运行容器

单独运行应用镜像时不会自带数据库；如果不传 `DATABASE_URL`，后端只使用内存演示数据：

```bash
docker run --rm -p 8080:80 bc-platform:local
```

如果要连接外部 PostgreSQL：

```bash
docker run --rm -p 8080:80 \
  -e DATABASE_URL='postgres://bc:bc_password@<db-host>:5432/bc' \
  bc-platform:local
```

访问地址：

- 管理后台：`http://127.0.0.1:8080/`
- 健康检查：`http://127.0.0.1:8080/api/health`

容器日志默认只保留后端服务输出和必要启动提示；Nginx 访问日志已关闭，Nginx 错误日志不会写入 Docker stdout/stderr，避免 `docker logs` 被前端静态资源和代理请求刷屏。后端启动等待健康检查期间会持续输出中文进度日志，包含已等待时间、剩余时间、curl 退出码、HTTP 状态、最近一次健康检查错误或响应片段，以及后端进程状态，便于排查数据库连接、端口监听或后端进程提前退出问题。

### 使用 Docker Compose

`docker-compose.yml` 会同时启动应用和 PostgreSQL，并把应用的 `DATABASE_URL` 指向 Compose 内的 `postgres` 服务：

```bash
docker compose up --build
```

默认对外端口是 `8080`；如果本机端口被占用，可以改用：

```bash
APP_PORT=18080 docker compose up --build
```

Compose 模式会把 PostgreSQL 数据保存在 `bc_postgres_data` volume 中；当前已落地后台业务模块在配置 `DATABASE_URL` 后都会使用 PostgreSQL 业务表持久化，运行时不再使用 `state_documents`。

### GitHub Container Registry

`main` 分支推送后，GitHub Actions 会构建并推送 `sha-<提交短哈希>` 镜像标签到 GHCR；推送 `v*` Git tag 时，会额外生成同名版本镜像标签。生产部署不要使用 `latest`，必须明确指定本次要部署的提交标签或版本标签。

```bash
echo "$GITHUB_TOKEN" | docker login ghcr.io -u <github-user> --password-stdin
docker pull ghcr.io/sydneypoole/bc:sha-<提交短哈希>
docker run --rm -p 8080:80 ghcr.io/sydneypoole/bc:sha-<提交短哈希>
```

如果要发布业务版本，先在 Git 中打版本标签，再推送标签触发镜像发布：

```bash
git tag v2026.06.23-1
git push origin v2026.06.23-1
docker pull ghcr.io/sydneypoole/bc:v2026.06.23-1
```

使用 Compose 部署远程镜像时，建议在服务器环境变量中写明版本：

```bash
export BC_IMAGE_TAG=sha-<提交短哈希>
export DATABASE_URL='postgresql://<user>:<password>@<host>:5432/<db>'
docker compose -f docker-compose.ghcr.yml pull
docker compose -f docker-compose.ghcr.yml up -d --force-recreate
```

`docker-compose.ghcr.yml` 只负责拉取 GHCR 中已经发布的应用镜像，不会在服务器上重新构建镜像；服务镜像固定写为 `ghcr.io/sydneypoole/bc:${BC_IMAGE_TAG}`，这样每次部署都能从日志和配置中直接看到具体版本。
如果仓库或 GHCR 包保持私有，部署机器需要把 `GITHUB_TOKEN` 设置为具备 `read:packages` 权限的 GitHub token。
当前 GitHub 打包流程会执行后端格式检查、后端类型检查和管理后台生产构建，但按发布效率要求跳过 `cargo test`；需要完整回归时请在本地或专门测试流程中单独运行后端测试。

### 环境变量

- `BACKEND_PORT`：容器内后端监听端口，默认 `8080`。
- `BACKEND_STARTUP_TIMEOUT_SECONDS`：入口脚本等待后端健康检查通过的最长秒数，默认 `60`。
- `BACKEND_STARTUP_LOG_INTERVAL_SECONDS`：入口脚本等待健康检查期间打印进度日志的间隔秒数，默认 `2`，必须大于 `0`。
- `RUST_LOG`：后端日志级别，默认 `info`。
- `DATABASE_URL`：可选，配置后后端使用 PostgreSQL 业务表持久化；未配置时使用内存演示数据。
- `DATABASE_MAX_CONNECTIONS`：PostgreSQL 最大连接池大小，默认 `30`。
- `DATABASE_MIN_CONNECTIONS`：PostgreSQL 最小连接池大小，默认 `2`。
- `DATABASE_ACQUIRE_TIMEOUT_SECONDS`：等待数据库连接的超时时间，默认 `10` 秒。
- `REDIS_URL`：可选，配置后启用 Redis 分布式锁和缓存失效能力；未配置时保持单进程本地行为。

当前镜像只包含应用本身，不包含 PostgreSQL。生产环境如果需要持久化，需要单独部署数据库并通过 `DATABASE_URL` 注入；本地 Compose 已提供 PostgreSQL 示例。
